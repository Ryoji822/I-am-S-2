# Task Module: UX Microcopy (短文UI文言)

## 使いどころ

ボタン文言、エラーメッセージ、確認文、通知、ツールチップなどの短い UI 文言を作るときに使う。読後に「迷わず操作できる・エラーを直せる」状態を作ることが目的。

## 構造

```
状態（成功/失敗/注意/確認）
→ 何が起きた（ユーザーに見える事実）
→ 何をすればよい（具体操作）
```

## テンプレート

```text
あなたはUIライターです。{画面}の{場所}に出す文言を作ります。目的は「迷わず操作できる／エラーを直せる」です。

[入力]
- 状態: {成功/失敗/注意/確認}
- 何が起きた: {ユーザーに見える事実}
- 原因の候補: {言える範囲。言えないなら空}
- 次にしてほしいこと: {具体操作}
- 制約: {文字数上限、改行可否}

[出力ルール]
- ラベルは動詞中心。明確さを最優先し、変に気の利いた表現は避ける。
- "再認しやすい表示"にして記憶負荷を減らす（必要情報はその場に置く）。
- エラー文は「何が起きた」→「どう直す」を必ず含める。
- 可能なら、入力欄の近くで出る想定で書く（遠いと探す負荷が上がる）。
```

## 設計原則

- 明確さが最優先。かわいさ、面白さ、ブランド感は後回し。
- 動詞中心のラベル。名詞化して曖昧にしない。
- 「再認」で済むように、必要情報をその場に置く。読者に記憶させない。
- エラー文は必ず「何が起きた」+「どう直す」のペア。
- 文字数制約があるなら、「どう直す」を優先し、「何が起きた」を短くする。

## good / bad pattern

### bad
パスワードが弱すぎます。もう一度入力してください。

### good
パスワードが短いです。8文字以上で作ってください。

違い: bad は何が問題かも、何文字必要かも分からない。good は事実と行動が明確。

## 状態別の基本パターン

### 成功
- 何が完了したか + 次にできること（任意）
- 例: 「保存しました」「送信しました。確認メールを送ります」

### 失敗
- 何が起きたか + どう直すか
- 例: 「メールアドレスの形式が違います。@を含む形式で入力してください」

### 注意
- 何が起きるか + 確認操作
- 例: 「この操作は取り消せません。削除しますか？」

### 確認
- 何をしようとしているか + 実行操作
- 例: 「3件のファイルをアップロードします」

## quality gate

- ユーザーが何をすべきか1文で分かるか
- 記憶に頼らず、再認で済むか
- 文字数制約を守っているか
- 動詞が主役になっているか
- 気の利いた表現で明確さを損なっていないか
