# Audience Module: General (組織非特定)

## 使いどころ

この module は、読み手が特定の組織に属さない場合、または Responses API / Claude Agent SDK 経由で汎用的にコンテンツを生成する場合に使う。

## 読み手モデルの推定手順

特定の audience module がない場合、次の順で読み手モデルを構築する。

1. source に書かれている前提条件から、想定読者の属性を推定する
2. 推定できない場合は、以下の3軸で仮説を置く
   - 専門知識レベル: 初学者 / 中級者 / 専門家
   - 判断レベル: 理解したい / 判断したい / 実行したい
   - テーマとの距離: 日常的 / 業務的 / 専門的

## スキーマ推定の原則

読み手の既存スキーマ（既知の枠組み）を推定し、それに接続する情報提示を設計する。

- **初学者**: 日常の経験に喩えてから専門概念を導入する。スキーマがないので作る。
- **中級者**: 既存スキーマを活性化し、それを更新する形で情報を提示する。
- **専門家**: 既存スキーマを前提とし、差分とトレードオフを前に出す。

## 入口の置き方

読み手の種別に応じて入口を変える。

### 理解したい読み手
- 読み手が体験したことのある「詰まり」から入る
- 「なぜそうなるのか」の問いを置く
- 平易な定義を先に出す

### 判断したい読み手
- 選択肢と比較軸を先に出す
- トレードオフを見せる
- 推奨案とリスクを分ける

### 実行したい読み手
- 「何をどう使うか」を先に出す
- 失敗しやすい例を見せる
- まず試す一歩を最後に置く

## 何を明示するか

次は省略しない。

- 読後に何が変わるか（行動、理解、判断）
- 前提となる知識（なければ定義する）
- 因果関係の橋
- 不確実なことは不確実だと言う
- 次にやること

## 情報構造の原則

旧情報→新情報の順序を守る。読者が既に知っている情報を先に提示してスキーマを活性化し、その上で新情報を導入する。

## quality gate

最後に次を確認する。

- 読み手の事前知識レベルに合っているか
- 目的に対して過不足がないか
- 旧情報→新情報の順序が守られているか
- 読み手が「要するに何か」を自分の言葉で言い直せるか
- 次に何をすべきかが残るか
