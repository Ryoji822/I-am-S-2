## 概要

AIが生成する「文法は整っているのに理解できない」文章は、ストーリーテリング不全として5つの層に分解できる。重要な前提: 「AIが悪い」のではなく「設計情報が欠けた指示が悪い」。

## 5層の構造

| 壊れる層 | AI症状（文は滑らか） | 読者に起きること | 指示での直し方 |
|---------|-------------------|---------------|-------------|
| **1. 目的・読者** | 何のための文章か曖昧、対象読者が揺れる | 「どの前提で読めばいいか」決められない | 「読後にできること」を1文で固定。読者の前提知識も明記 |
| **2. 背景・定義** | 用語が定義されない、主語や指示語が飛ぶ | 統合に失敗。指示語の不明確さで読みが崩壊 | 先に"前提・用語表"を置く。初出語は必ず短く説明 |
| **3. 構造（開始→結末）** | 段落が同列に並び、結論に収束しない | 因果の筋が見えず、偶然の寄せ集めに見える | Labov型（背景→進行→意味→結果→次の手）へ強制 |
| **4. 認知負荷** | 情報が多すぎる、余談と本筋が混ざる | 同時に扱える量の限界を超過（約4チャンク） | 「捨てる基準」と「分け方」を指示（1段落1役割、最大見出し数） |
| **5. 信頼（受容性）** | 自信満々な断言、根拠の出所が曖昧 | 「信じてよいか」判断ができず離脱 | 不確実な点は不確実と言う、根拠を分ける、検証手順を入れる |

## 各層の詳細と処方箋

### 層1: 目的・読者

最も上流の設計情報。ここが欠けると以降すべての判断基準が不在になる。

**根拠**: Web指針が「ユーザーは何を知りたいか」から始めよと言い、関連性理論が「読者にとっての認知効果」を基準とする。

**処方箋の実装**:
- audience module を必ず選択する
- 「読後にできること」を input contract の goal に明記する
- 読者の前提知識を3段階（知っている / 聞いたことはある / 知らない）で分類

### 層2: 背景・定義

読者のスキーマ活性化に必要な最小限の前提情報。

**根拠**: Kintsch の構築統合モデル。構築段階で必要な知識が活性化されなければ統合段階に進めない。可読性は語の長さだけでなく前提知識の有無でも決まる。

**処方箋の実装**:
- 初出の専門用語に必ず1文の定義を付ける
- 指示語（それ、これ、この問題）の先行詞が同一段落内に存在することを確認
- 旧情報→新情報の順序を守る

### 層3: 構造

因果の筋が通った流れ。偶然の並列ではなく必然の連鎖。

**根拠**: アリストテレス「偶然に始まり偶然に終わるものは悪い」。Labov の語り構造（概要→背景→進行→評価→結果→結末）。RST（修辞構造理論）の核と衛星の階層。

**処方箋の実装**:
- Labov型の6機能要素を明示的にマッピング
- 各段落の役割（理由 / 具体化 / 対比 / 手順）を設計時に確定
- セクション間を「だからこそ」「しかしここで」で接続（post hoc の排除）

### 層4: 認知負荷

ワーキングメモリの容量制約（約4チャンク）に適合する情報量と構造。

**根拠**: Cowan のワーキングメモリ容量。Mayer の12原則（coherence principle: 余計な情報を減らすほど学びやすい）。Sweller の認知負荷理論。

**処方箋の実装**:
- 1段落1アイデア
- 見出し数は最大7±2（ミラーの法則）
- 余談・歴史・網羅的背景は core/support/develop トリアージで分離
- 「捨てる基準」をプロンプトに明記: スルーラインに接続しない情報は削除

### 層5: 信頼

読者が「この情報を受け入れてよいか」を判断できる設計。

**根拠**: LLM のハルシネーション（不確かでも当てにいく圧力）。グライスの質の格率（十分な証拠のないことを言うな）。確信度キャリブレーション問題。

**処方箋の実装**:
- 事実と推測を明示的に分離（「〜である」vs「〜と考えられる」）
- 根拠の出所を明記
- 不確実な点は「不確実」と述べる
- 条件つき主張の断定化を禁止（ハルシネーション対策）

## 層間の依存関係

```
層1: 目的・読者 ──→ すべての判断基準
       ↓
層2: 背景・定義 ──→ 理解の前提条件
       ↓
層3: 構造 ──→ 因果の骨格
       ↓
層4: 認知負荷 ──→ 処理可能性
       ↓
層5: 信頼 ──→ 受容性
```

上流の層が壊れていると、下流をいくら修正しても効果がない。層1（目的・読者）が不在のまま層3（構造）を整えても、何のための構造か不明なため機能しない。

## 抽象フレームの7要素

5層分析から導出される「理解しやすさの最小単位」:

| 要素 | 定義 | 失敗時 | 優先度 |
|------|------|--------|--------|
| **読者と目的** | 誰が読んで、読後に何ができるべきか | 散漫 | 最優先（30%） |
| **中核メッセージ** | 1〜2文で言える答え | 収束しない | 高（20%） |
| **背景（定義含む）** | 時間・場所・前提・用語の最小セット | 統合失敗 | 高（20%） |
| **進行（問題→解決）** | 問い/問題がどう進みどう解決するか | 因果不明 | 中（15%） |
| **意味づけ** | なぜ重要か、読む理由 | 「ふーん」で終了 | 中（10%） |
| **次の手** | 読者が次にとる行動 | 行動に繋がらない | 中（5%） |
| **負荷管理** | 不要情報を捨て、分け、合図する | 途中で迷子 | 横串（常時） |

## 要素間の関係

```
読者と目的 → 中核メッセージ
    ↓              ↓
前提・用語 → 構造（背景→問題→解決→意味→次の手）
                   ↓
         段落の関係づけ（理由/具体化/対比/手順）
                   ↓
              負荷管理（捨てる・分ける・合図）
                   ↓
              理解度テスト → 構造にフィードバック
```

## 実装への変換

1. 指示設計時に層1（目的・読者）を必ず最初に固定する
2. 7要素のうち少なくとも上位4つ（読者・メッセージ・背景・進行）を input contract に含める
3. draft 完了後、5層を上から順に検査する（上流の不備が見つかったら下流の検査を中断し上流を修正）
4. evaluation-rubric（31）の「audience fit」「structural clarity」「causal continuity」が各層に対応
5. 層4の負荷管理は anti-slop（30）の冗長性バイアス対策と連動する
