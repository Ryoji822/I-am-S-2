# I-am-S-2 現行パイプライン調査メモ

## 現行構成の骨格

GitHub Actionsは `daily-intelligence.yml` により毎日08:00 JST相当で実行され、`scripts/run-pipeline.sh` を通じて、収集、X投稿注入、Blue分析、Red Team、Arbiter、Static Intelligence更新、日次レポート、検証、Git commit/push、Slack通知を直列実行している。OpenCodeは `zai/glm-5` を単一モデルとして使い、Firecrawl MCPを情報収集手段として有効化している。

## 既存設計の強み

現行設計は、単なるニュース要約ではなく、PIR/KIQ、仮説、シナリオ、I&W、確度、事実と判断の分離、Red Team、Arbiter、Static Intelligence更新という意思決定支援型の骨格を既に持っている。特に `pirs.json` はビジネス意思決定の問いを中心に構成され、`collection_plan.json` は各KIQに対する検索クエリ、ソース種別、頻度、最小ソース数を定義している。`phase6-report.md` は当日差分、事実と判断の分離、確度表記、行動提言を強く要求しており、最終成果物の方向性は正しい。

## 主な不足

不足は「軍事用語の不足」ではなく、情報精査プロセスの工程密度不足である。現行は `COLLECT -> ANALYZE -> RED TEAM -> ARBITER -> STATIC UPDATE -> REPORT` の大きな直列段階に圧縮されており、収集前の要求管理、収集計画の優先順位付け、ソース評価、情報正規化、欠落検出、反証収集、仮説別証拠マトリクス、確度校正、状態変更の監査証跡が独立したゲートとして十分に分かれていない。

また、失敗時フォールバックが前日コピーやBlue出力の素通しを許すため、DEGRADED表示はあるものの、意思決定に使ってはいけない状態の遮断が弱い。検証スクリプトはファイル存在、最低件数、見出し、リンク形式、確度有無などの表層検査が中心であり、PIR/KIQ充足、ソース独立性、Admiralty評価の整合、仮説・シナリオ・指標IDの参照整合、判断変更の根拠十分性までは厳密に検査していない。

## 具体的な状態管理ギャップ

`hypotheses.json` は永続的な仮説定義と日次Arbiterの長文判断履歴が同居しており、構造定義とランニング評価が混ざっている。`scenarios.json` は多くの `IND-*` を参照しているが、`indicators.json` 側の実定義は限定的であり、参照整合性の不足がある。`arbiter_log.json` は監査ログの思想を持つが、現行パイプラインでは判断変更の全履歴を強制的に追記するゲートとして前面化されていない。

## 再設計で維持すべき前提

ユーザー要望に従い、3冊の教範は軍事運用へ直訳せず、ビジネスのための意思決定支援にアナロジーする。維持すべき中核は、PIR/KIQで意思決定者の問いを明確にし、収集を問いに従属させ、情報をソース評価・妥当性確認・反証・仮説照合・確度校正を通じてIntelligenceへ変換し、意思決定者へ配布し、フィードバックで次の収集要求に戻す循環である。
