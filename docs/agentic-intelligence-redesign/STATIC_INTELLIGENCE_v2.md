# Static Intelligence v2 仕様

`docs/agentic-intelligence-redesign/README.md` §6 / §8 / §12 と、`prompts/phase6-report.md` の Daily 6要素分離(事実/仮定/判断/不確実性/含意/次回収集)を、**時間スケールが長い** Static Intelligence の性質に合わせて圧縮した構造仕様。

## なぜv2が要るか

旧 `static_intelligence/*.md` は次の癖を持っていた。

- 確度%は書くが、その根拠は外部 (`state/arbiter-*.md` v3.66 等) にしかない
- C-I (Confirmed-Insufficient) 表記が読者に解釈を丸投げする
- 「逆説」「対立軸」表現で事実と判断が混ざる
- 規範的言及 (「こう進むべき」「逆転可能性が現実的」) が予測と混在
- 中国市場のように観測根拠が薄い場合でも、確度%だけが残り、情報非対称性が見えない

Daily Report 側を v2 (6要素分離) にしても、参照される Static が旧構造のままだと、Daily が引用するたびに旧構造の重力に引きずられる。Static を v2 化することで、Daily ↔ Static の往復が同一思想で回る。

## v2 ファイル構造

すべての企業ファイル (`anthropic.md` / `openai.md` / `google.md` / `xai.md` / `bytedance.md`) は次の7セクション + 付録で構成する。

```markdown
# {会社名}

> 最終判断更新: YYYY-MM-DD
> 全体確信度: 高 | 中 | 低
> 情報非対称性: (該当時のみ。例: 「中国市場の透明性低により観測根拠は限定的」)
> 主参照: hypotheses.json#H-XXX, indicators.json#IND-XXX

## 0. 一文要約

我々はこの企業を「A」と読んでいる。最大の根拠は「B」、もし「C」が観測されたら判断は変わる。

(1〜2段落、30秒で読める粒度。意思決定者がここだけ読んでも次に動ける状態にする)

## 1. コア判断

我々の現在の解釈をストーリーで1〜3段落。

書き始めは判断と確信度。続いて、その判断に至った因果の橋。事実の羅列ではなく、「だからこう読んでいる」という解釈の重心が最初に立つこと。

## 2. 判断の重心 (Load-Bearing Reasoning)

このコア判断を支える、なくなったら判断が崩れる根拠を3〜5項目。

| 重要度 | 観測した事実 | この判断との関係 | 信頼度 | 参照 |
|:-:|---|---|:-:|---|
| 高 | (具体的観測。具体数値必須) | (なぜこれがコア判断を支えるか) | A-3 | [INFO-XXX] / [EVD-...] |

「重要度」は、その項目が崩れた場合にコア判断がどれだけ揺らぐかで決める。「信頼度」は Admiralty Code (A-3 等)、または「中国市場のため低」のような言語的指示。

## 3. 反証の閾値 (Falsification Triggers)

「何が観測されたら、現在のコア判断が間違いだと認めるか」を具体的に3〜5項目。

| 反証指標 | 観測したら何が崩れるか | 期限 | 監視先 |
|---|---|:-:|---|
| (具体的事実) | (どの判断/仮説/重心が崩れるか) | 30日 | [IND-XXX] |

ここに書ける項目があるかどうかが、コア判断が反証可能 (Falsifiable) か単なる印象かを切り分ける。書けないなら判断自体を見直す。

## 4. 進行中の仮説 (Active Hypotheses)

| 仮説ID | 一文要約 | 確度 | 確度の根拠 | 強める証拠 | 弱める証拠 |
|---|---|:-:|---|---|---|
| H-XXX-001 | (一文) | 52% | (なぜ52%か。独立裏付け数、矛盾、ソース信頼性) | [INFO-...] | [INFO-...] |

確度の根拠を1行で書く。書けないなら確度を低くするか、`不明` と明示する。`[Arbiter v3.66 参照]` のような外部ポインタは付録の出典欄にだけ書き、本表に持ち込まない。

## 5. 監視指標 (Watch List)

| 指標ID | 何を見るか | 閾値 | 現在値 | 最終確認 |
|---|---|---|---|:-:|
| [IND-XXX] | (定義) | (発火条件) | (直近観測) | YYYY-MM-DD |

`config/indicators.json` への参照と現在値を統合した運用面。

## 6. 変化履歴 (Decision History)

| 日付 | 変更 | きっかけ | 過去 → 現在 |
|:-:|---|---|---|
| YYYY-MM-DD | コア判断改訂 | [INFO-XXX] | 「~~~」 → 「~~~」 |

判断が変わった節目だけ。日次の確度微変動は記録しない (それは `state/arbiter-*.md` の役割)。

## 7. ブラインドスポット (Known Unknowns)

我々が見えていない・見ようとしていない側面を3〜5項目、箇条書きで。

- (我々の視点が偏っている可能性のある領域)
- (情報を取れていない理由を含む)

書き手の自己批判的視点を残すための区画。Red的な思考を構造化する。

## 付録: 直近30日の参照Evidence

| Evidence | 用途 | 出典 |
|---|---|---|
| [INFO-XXX] / [EVD-...] | 本ファイルでの引用箇所 | URL or 内部参照 |

外部参照 (`[Arbiter v3.XX]`, 過去日 INFO 等) はここにのみ集約する。本文を圧縮した分の出典追跡は付録で担保する。
```

## 横断ファイル (`market-overview.md`)

企業構造ではなく市場全体を扱うため、§1〜§7 を市場視点に置き換える。

- §0 一文要約 (市場全体について)
- §1 コア判断 (市場の現在の構図)
- §2 判断の重心 (市場構造を支える観測)
- §3 反証の閾値 (構図が崩れる条件)
- §4 進行中の仮説 (市場レベルの仮説)
- §5 監視指標 (市場 I&W)
- §6 変化履歴 (構図の節目)
- §7 ブラインドスポット (市場全体に対する盲点)

冒頭に「プレイヤー一覧スナップショット」表を §0 の前に置く。

## 横断ファイル (`scenario-tracker.md`)

シナリオごとに同じ7セクションのミニ版を適用する。

- 全体冒頭: 確率推移サマリ表 (4シナリオ + Black Swan、過去20日)
- シナリオ単位:
  - §0 一文要約
  - §1 コア判断
  - §2 判断の重心
  - §3 反証の閾値 (= シナリオ確率を下げる条件)
  - §4 仮説 (シナリオを構成する仮説群)
  - §5 監視指標
  - §6 変化履歴 (確率の主要変動)
  - §7 ブラインドスポット

## Daily ↔ Static の対応

`prompts/phase6-report.md` の Daily 6要素 ⇄ Static v2 の対応表。

| Daily §1.x の要素 | Static v2 の対応セクション |
|---|---|
| 事実 (Facts) | §2 判断の重心 (の観測が累積した状態) |
| 仮定 (Assumptions) | §1 コア判断 (の前提として埋め込まれている) |
| 判断 (Judgment) | §0 一文要約 + §1 コア判断 |
| 不確実性 (Uncertainty) | §3 反証の閾値 + §7 ブラインドスポット |
| 意思決定への含意 | §0 + §1 (継続的な含意は本ファイルが永続化) |
| 次回収集要求 | §3 反証の閾値 (の監視先 IND-XXX) |

Daily が日次で動かすのは「確度・観測・反証指標の現在値」だけで、Static の構造そのものは判断変更時にだけ更新する。

## 文体規律

reframing-storytelling skill の anti-slop / 35-japanese-plain / 39-humanizer-ja に従う。特に重要なもの:

- 「〜として位置づけられる」「〜の重要な転換点」のような意義の上乗せをしない
- 「また」「さらに」「加えて」を等間隔配置しない
- emダッシュ (——) は使わない。読点 or 括弧で代替
- 「〜と言えるでしょう」「〜と考えられます」で全結論を締めない。確信があれば言い切る
- 連用中止形 (〜し、〜し、〜する) を3つ以上連ねない
- 太字+コロンの箇条書き (`- **項目:** 説明`) を §1 / §0 で使わない (表に逃げない)
- 「重要」「画期的」「圧巻」「魅力的」などの形容詞インフレを排除
- 「今後ますます重要になる」「期待される」で締めない

## 検証 (validate-output.sh への追加項目, WARN レベル)

新セクション「Static Intelligence v2 Structure」を追加し、各 `.md` について以下を確認:

- 必須見出し7セクション (`## 0.` 〜 `## 7.`) + `## 付録` が揃っているか
- ヘッダーに `> 最終判断更新:` `> 全体確信度:` の2行があるか
- §1 コア判断が表ではなく段落で書かれているか
- §2 が表で、列に「重要度」「観測」「この判断との関係」「信頼度」「参照」を含むか
- §4 仮説表に「確度の根拠」列があるか
- 全体で `[Arbiter v3.\d+]` のような外部スナップショット参照が本文 (§0〜§7) に含まれていないか (付録のみ可)

すべて WARN レベル。1週間安定後 FAIL に昇格。

## ロールバック

旧ファイルは git history に残るので、問題があれば revert で戻せる。Daily Report (`Intelligence/*.md`) は v1/v2 のどちらの Static も読める形 (引用は INFO-XXX / IND-XXX 形式で抽象化されている) なので、Static を v2 化しても Daily 出力は壊れない。
